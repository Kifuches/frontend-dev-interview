# Change Detection

### Кратко

`Change Detection` - механизм обнаружения изменений в Angular. Использует `Zone.js` для перехвата асинхронных операций (events, setTimeout, HTTP). При изменениях проходит по дереву компонентов сверху вниз, проверяя bindings. Стратегии: `Default` (проверяет всегда) и `OnPush` (проверяет только при изменении `@Input` или события). `OnPush` оптимизирует производительность.

`OnPush` + `Immutable data` = значительное улучшение производительности больших приложений.

---

`Change Detection` — механизм Angular, который проверяет, изменились ли данные компонента, и обновляет DOM только там, где изменились привязки.

Angular не перерисовывает весь экран заново.

Он проверяет bindings:

```ts
{{ user.name }}
[disabled]="isLoading"
[class.active]="selected"
```

и сравнивает старое значение с новым.

#### **Как это выглядит**

```ts
Пользователь нажал кнопку
↓
Изменилось состояние
↓
Angular запускает Change Detection
↓
Проверяет дерево компонентов
↓
Сравнивает значения в шаблонах
↓
Обновляет DOM
```

Например:

```ts
name = 'John';

changeName() {
    this.name = 'Mike';
}
```

В шаблоне:

```html
{{ name }}
```

Angular помнит старое значение `John`, после события видит новое значение `Mike` и обновляет текст в DOM.

### Что запускает `Change Detection`?

Чаще всего `Change Detection` запускается после асинхронных событий.

Например:

```ts
click
input
setTimeout
setInterval
Promise
HTTP request
Router navigation
AsyncPipe
Signal update
```

В классическом Angular это обычно происходит через `Zone.js`.

`Zone.js` перехватывает асинхронную операцию и сообщает Angular:

“Возможно, данные изменились.”

После этого Angular запускает проверку.

**Важно**

`Zone.js` не знает, что именно изменилось.

Он просто говорит Angular, что что-то могло измениться.

Поэтому Angular должен проверить компоненты.

## Default

При стратегии `Default` Angular проходит по дереву компонентов после асинхронных событий: кликов, таймеров, HTTP-ответов и других задач, которые отслеживает Zone.js.

При любом событии Angular проходит по дереву компонентов и проверяет этот компонент тоже.
Это проще, но на больших экранах может быть дороже.

## OnPush

`OnPush` ограничивает проверки компонента. Компонент обычно проверяется, когда:

- изменился reference у `@Input`;
- произошло событие внутри компонента;
- сработал `async` pipe;
- изменился `Signal`, который используется в шаблоне;
- код явно вызвал `markForCheck` или `detectChanges`.

#### Почему `OnPush` может не обновить шаблон?

Частая причина — мутация объекта без изменения ссылки.

Например:

```ts
@Input() user!: User;

changeUser() {
    this.user.name = 'Mike';
}
```

Для Angular ссылка на `user` осталась той же самой, поэтому `OnPush` компонент может не обновиться.

Правильно:

```ts
changeUser() {
    this.user = {
        ...this.user,
        name: 'Mike'
    };
}
```

#### Почему массив не обновился?

Потому что сделали мутацию:

```ts
this.items.push(newItem);
```

А надо создать новый массив:

```ts
this.items = [...this.items, newItem];
```

## Что проверяют на интервью

Интервьюеру важно увидеть, что ты не воспринимаешь `OnPush` как магическую оптимизацию. Нужно понимать reference equality, иммутабельные обновления, работу `async` pipe и ситуации, где лишние подписки или мутации ломают ожидаемое поведение UI.
